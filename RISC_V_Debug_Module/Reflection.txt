Q1 — Why does the spec mandate dmactive resets everything else?
dmactive acts as a master enable for the entire Debug Module. When it is deasserted, the DM is in an undefined or partially-powered state — the host SoC may be in reset itself. If registers like dmcontrol or command retained stale values across a power cycle or warm reset, a debugger reconnecting could accidentally re-issue a halt or abstract command against a hart that never consented to it. Keeping everything frozen at dmactive=0 gives the debugger a guaranteed clean slate on every connection.

Q2 — What breaks if abstractcs.busy isn't set during EXEC?
The debugger polls abstractcs.busy to know when it is safe to read data0. If busy is never set, the debugger sees busy=0 immediately and reads data0 before the FSM has finished fetching the register from the hart. It gets stale or zero data. Worse, if the debugger issues a second command before the first finishes, the FSM has two commands in flight simultaneously — cmderr encoding becomes ambiguous and the hart handshake can deadlock.

Q3 — How to extend to multi-hart with hartsel?
dmcontrol[25:16] is the hartsel field (hart select index). Currently the DM only drives one halt/resume line and reads one halted/running. For multi-hart you need an array — halt[N], halted[N], etc. The halt-resume controller reads hartsel and indexes into these arrays. dmstatus.allhalted becomes an AND reduction across all selected harts, and anyhalted becomes OR. The abstract command FSM similarly routes hart_req_valid and reads hart_resp_rdata from the hart indexed by hartsel.

Q4 — Why does halt win over resume in the spec?
Safety. If a debugger issues both in the same cycle — which can happen due to race conditions in OpenOCD or GDB — the ambiguous outcome of resuming an unprepared hart is far more dangerous than keeping it halted. A halted hart is inert; a spuriously resumed hart executes arbitrary instructions from wherever dpc points, which can corrupt architectural state, trigger interrupts, or damage hardware peripherals. The spec resolves the ambiguity conservatively: halt always wins.

Q5 — Where is the DMI-to-DM clock crossing and how to cross it safely?
The DMI runs on the JTAG TCK clock (typically 1–10 MHz), while the DM runs on the SoC core clock (potentially hundreds of MHz) — these are completely asynchronous. The crossing sits at the boundary between the DTM (Debug Transport Module, on the JTAG side) and the DMI slave (on the SoC side). The standard safe crossing method is a req/ack handshake with two-flop synchronisers on both the request and acknowledge paths. The DTM asserts dmi_req_valid and holds all request signals stable; the DM samples them after two sync flops, processes the transaction, then pulses dmi_resp_valid back through its own two-flop synchroniser to the TCK domain. No data changes while the handshake is in flight, so metastability only affects the valid/ack control signals, not the address or data buses.
